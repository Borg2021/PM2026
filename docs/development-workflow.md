## 1. 总览

```
你提需求
  │
  ▼
┌─────────────────────────────────────────────────┐
│ STEP 1  写 Spec    docs/specs/YYYY-MM-DD-xxx.md │
│         5-10 分钟   需求 + 验收标准 + 影响范围      │
├─────────────────────────────────────────────────┤
│ STEP 2  AI 出 Plan  AI 读 Spec → 探索代码         │
│         2-5 分钟    → 输出涉及文件 + 变更顺序       │
│         你审核                                     │
├─────────────────────────────────────────────────┤
│ STEP 3  分层实现   自底向上，每层编译通过再继续      │
│         主体时间    Domain → App → Infra → API → FE │
├─────────────────────────────────────────────────┤
│ STEP 4  验证       启动系统 → 对照 Spec 逐条验收    │
│         5-10 分钟   不通过 → 回到 STEP 3            │
├─────────────────────────────────────────────────┤
│ STEP 5  提交       git commit 结构化消息           │
│         1 分钟      关联 Spec 文件                  │
└─────────────────────────────────────────────────┘
```

### 核心理念

| 原则 | 说明 |
|-----|------|
| **先想再写** | Spec 没写清楚前不碰代码 |
| **自底向上** | Domain → Application → Infrastructure → API → Frontend |
| **逐层验证** | 每层改完 → 编译通过 → 再下一层 |
| **最小变更** | 只改必须改的文件，diff 越小越好 |
| **验收驱动** | Spec 里的验收标准就是"改完的证明" |

## 2. STEP 1 — 写 Spec（需求定义）

### 2.1 放在哪里

```
项目管理系统V1/
└── docs/
    └── specs/                          ← 所有 Spec 放这里
        └── 2026-06-17-添加项目备注字段.md  ← 日期 + 功能名
```

### 2.2 Spec 模板

```markdown
# [功能名称]

> 日期：YYYY-MM-DD　|　状态：草稿 / 已实现

## 1. 为什么要做（背景）

一两句话说清楚：现在有什么问题？为什么要改？

## 2. 要达成什么（需求）

- [ ] 需求点 1
- [ ] 需求点 2
- [ ] 需求点 3

## 3. 怎么算做完（验收标准）

> 每条标准必须可验证，忌"功能正常"这种模糊描述

- [ ] 场景 A：操作步骤 → 预期结果
- [ ] 场景 B：操作步骤 → 预期结果
- [ ] 异常场景：错误输入 → 预期报错信息

## 4. 影响范围（初判）

> 不确定没关系，写个大概，AI 会在 STEP 2 帮你补全

- 涉及层：Domain / Application / Infrastructure / API / Frontend
- 涉及实体：Project、User ...
- 涉及页面：ProjectEditor.vue ...

## 5. 备注

截图、参考资料、特殊说明
```

### 2.3 好 Spec vs 坏 Spec

| 坏 Spec ❌ | 好 Spec ✅ |
|-----------|-----------|
| "给项目加个备注功能" | "项目编辑页面增加一个备注字段，500 字上限，所有角色可见，保存时记录操作日志" |
| "列表要快点" | "项目列表 API 响应时间从 2s 降到 500ms 以内，列表页打开后 1s 内显示数据" |
| "修复文件上传 bug" | "上传 50MB 文件时浏览器报 413 错误，期望能上传成功（系统限制 100MB）" |

### 2.4 什么时候可以跳过 Spec

| 场景 | 是否可跳过 |
|-----|-----------|
| 修一个明确的 bug（报错信息清晰、原因一眼可见） | ✅ 跳过，直接改 |
| 改一个文案 / 调整样式 | ✅ 跳过，直接改 |
| 新增一个接口 / 字段 / 页面 | ❌ 必须写 |
| 改业务规则（状态流转、权限逻辑） | ❌ 必须写 |
| 涉及 3 个以上文件 | ❌ 必须写 |

## 3. STEP 2 — AI 出 Plan（实施计划）

### 3.1 如何让 AI 出 Plan

把 Spec 文件路径告诉 AI，它会自己读 → 探索代码 → 输出计划：

```
"请根据 docs/specs/2026-06-17-xxx.md，探索代码后出实施计划"
```

AI 会自动：
1. 读 Spec 理解需求
2. 搜索现有代码（实体、Handler、Controller、页面）
3. 列出所有需要改的文件
4. 给出分层变更顺序

### 3.2 Plan 输出格式（AI 自动生成）

```
## 实施计划

### 涉及文件
| 层 | 文件 | 操作 |
|----|------|------|
| Domain | Entities/Project.cs | 新增属性 |
| Application | DTOs/ProjectDto.cs | 新增映射字段 |
| API | Controllers/ProjectController.cs | 无需改动 |
| Frontend | views/ProjectEditor.vue | 新增表单项 |

### 变更顺序
1. Domain: Project.cs — 添加 Xxx 属性
2. Application: ProjectDto.cs — 添加映射
3. Frontend: ProjectEditor.vue — 添加 UI

### 验证方式
- 编译通过
- 项目详情 API 返回新字段
- 编辑页面显示新字段
```

### 3.3 你审核什么

| 检查项 | 问自己 |
|-------|--------|
| 文件清单 | 有没有漏？有没有多余的？ |
| 变更顺序 | 是否自底向上？ |
| 影响范围 | 会不会影响到不相关的功能？ |
| 验证方式 | 能不能实际执行？ |

> 审核通过后说 "开始实现"，AI 进入 STEP 3。

## 4. STEP 3 — 分层实现（编码）

### 4.1 变更顺序（铁律）

```
后端：Domain → Application → Infrastructure → API

前端：Types → API 层 → Store → Views
```

**为什么必须自底向上？**

- Domain 改了实体 → Application 的 DTO 才能跟着改
- Application 改了接口 → API 的 Controller 才能知道返回什么
- 后端 API 定了 → 前端才能知道数据结构

### 4.2 后端各层的改法

```
         ┌──────────────────────────────────┐
  ①      │  Domain                           │
         │  Entities/    新增/修改实体属性     │
         │  Enums/       新增枚举值           │
         │  Interfaces/  新增仓储方法签名      │
         └──────────────┬───────────────────┘
                        │ dotnet build ✅
                        ▼
         ┌──────────────────────────────────┐
  ②      │  Application                      │
         │  Commands/   新增/修改 CQRS Handler│
         │  Queries/    修改查询逻辑          │
         │  DTOs/       新增 DTO 或字段       │
         │  Validators/ 新增校验规则          │
         └──────────────┬───────────────────┘
                        │ dotnet build ✅
                        ▼
         ┌──────────────────────────────────┐
  ③      │  Infrastructure                   │
         │  Data/       DbContext 配置变更    │
         │  Repositories/ 仓储实现调整        │
         └──────────────┬───────────────────┘
                        │ dotnet build ✅
                        ▼
         ┌──────────────────────────────────┐
  ④      │  API                              │
         │  Controllers/ 路由、参数、权限调整  │
         │  Auth/        新增权限码           │
         └──────────────────────────────────┘
                        │ dotnet build ✅
```

### 4.3 前端的改法

```
  ⑤      ┌──────────────────────────────────┐
         │  types/      新增/修改 TS 接口     │
         └──────────────┬───────────────────┘
                        ▼
         ┌──────────────────────────────────┐
         │  api/        新增/修改 API 调用    │
         └──────────────┬───────────────────┘
                        ▼
         ┌──────────────────────────────────┐
         │  store/      如需全局状态（少见）   │
         └──────────────┬───────────────────┘
                        ▼
         ┌──────────────────────────────────┐
         │  views/      页面 UI 变更          │
         │  components/ 子组件调整            │
         └──────────────────────────────────┘
                        │ npm run build ✅
```

### 4.4 AI 协作要点

| 做法 | 原因 |
|-----|------|
| 一次只让 AI 改一层 | 避免 AI 一次输出太多而出错 |
| 改完一层说 "编译通过，继续下一层" | AI 知道当前状态，不会重复或跳过 |
| 不要让 AI 自动格式化 | 按 CLAUDE.md 第 3 条，不自动格式化 |
| 改完前端让 AI 自己启动验证 | AI 用 run skill 启动系统做端到端验证 |

## 5. STEP 4 — 验证（验收）

### 5.1 验证清单

拿出 STEP 1 写的 Spec，逐条对照验收标准：

```
┌─────────────────────────────────────────────────┐
│  验收标准逐条验证                                 │
│                                                  │
│  □ 场景 A: 操作 → 结果                           │
│  □ 场景 B: 操作 → 结果                           │
│  □ 异常场景: 操作 → 预期报错                      │
│  □ 编译通过 (dotnet build + npm run build)       │
│  □ 不破坏现有功能（核心页面快速抽查）              │
└─────────────────────────────────────────────────┘
```

### 5.2 验证方式

| 类型 | 方法 |
|-----|------|
| 后端 API | `curl` 或 Swagger 直接调接口 |
| 前端 UI | 浏览器打开 `https://localhost:9091`，实际操作一遍 |
| 数据库 | `sqlcmd` 或 VS Code 插件查表验证 |
| 编译 | `dotnet build` / `npm run build` |

### 5.3 快速冒烟（不改也要验）

即使改动很小，也快速抽查这些核心页面：

| 页面 | 路由 | 验证点 |
|-----|------|-------|
| 登录 | `/login` | 能正常登录 |
| 项目列表 | `/project/list` | 列表加载、搜索可用 |
| 项目详情 | `/project/view/:id` | 详情页正常打开 |
| 工作台 | `/workbench` | 待办数据正常 |

> 验证不通过 → 回到 STEP 3 修改 → 再验证 → 直到全部通过。

## 6. STEP 5 — 提交（版本记录）

### 6.1 Commit 消息格式

```
<类型>: <简短描述>

<详细说明（可选）>

关联: docs/specs/2026-06-17-xxx.md
```

**类型**：

| 前缀 | 用途 |
|-----|------|
| `feat:` | 新功能 |
| `fix:` | Bug 修复 |
| `refactor:` | 重构（不改功能） |
| `docs:` | 文档变更 |
| `style:` | 样式调整 |

### 6.2 示例

```
feat: 项目新增备注字段

- Domain: Project 实体增加 Remark 属性
- Application: ProjectDto 增加映射
- Frontend: ProjectEditor.vue 增加备注输入框
- 字段脱敏：需 project:view:remark 权限

关联: docs/specs/2026-06-17-添加项目备注字段.md
```

### 6.3 提交时机

```
每完成一个独立 Spec → 一次 commit
不要一个 commit 包含多个不相关的变更
```

## 7. 完整示例

> 假设需求："项目列表页增加按客户名称搜索"

### 7.1 Spec 文件

`docs/specs/2026-06-17-项目列表增加客户名称搜索.md`

```markdown
# 项目列表页增加按客户名称搜索

> 日期：2026-06-17　|　状态：已实现

## 1. 为什么要做

目前项目列表只能按项目编号、名称、PM 搜索，无法按客户名称定位项目。

## 2. 要达成什么

- [x] 项目列表搜索栏增加"客户名称"输入框
- [x] 支持模糊匹配
- [x] 与其他搜索条件可组合使用

## 3. 怎么算做完

- [x] 输入客户名称 "华为" → 列表只显示客户名称含"华为"的项目
- [x] 同时输入编号 "PRJ" + 客户名称 "华为" → 两个条件同时生效
- [x] 清空客户名称 → 恢复显示全部项目
- [x] 输入不存在的客户名 → 列表显示 "暂无数据"

## 4. 影响范围（初判）

- 涉及层：Application、Frontend
- 涉及文件：ProjectQueries、ProjectList.vue
```

### 7.2 AI 出 Plan

```
"请根据 docs/specs/2026-06-17-项目列表增加客户名称搜索.md 出实施计划"
```

AI 探索代码后输出：

```
## 实施计划

### 涉及文件
| 层 | 文件 | 操作 |
|----|------|------|
| Application | Projects/Queries/GetProjectListQuery.cs | 增加 customerName 参数 + 过滤逻辑 |
| Frontend | views/ProjectList.vue | 搜索栏增加客户名称输入框 |
| Frontend | api/project.ts | 请求参数增加 customerName |

### 变更顺序
1. Application: GetProjectListQuery.cs — 增加参数和过滤
2. Frontend: api/project.ts — 请求参数增加字段
3. Frontend: ProjectList.vue — UI 增加搜索框

### 验证
- 编译通过
- curl 测试 API：带 customerName 参数查询
- 浏览器操作：输入客户名 → 列表过滤
```

### 7.3 分层实现

```
你: 开始实现，从 Application 层开始

AI: [改 GetProjectListQuery.cs，增加 customerName 参数]
    → dotnet build ✅

你: 编译通过，继续前端

AI: [改 project.ts + ProjectList.vue]
    → npm run build ✅
```

### 7.4 验证

```
你: 启动系统，帮我验证

AI: [启动 dotnet run]
    → curl API 验证 ✅
    → 打开浏览器操作验证 ✅
    全部验收通过 ✅
```

### 7.5 提交

```
feat: 项目列表增加客户名称搜索

- GetProjectListQuery: 增加 customerName 模糊搜索参数
- ProjectList.vue: 搜索栏增加客户名称输入框
- project.ts: 请求参数增加 customerName

关联: docs/specs/2026-06-17-项目列表增加客户名称搜索.md
```

## 8. AI 协作技巧

### 8.1 给 AI 的指令模式

| 阶段 | 推荐说法 |
|-----|---------|
| 启动 | "请根据 `docs/specs/xxx.md` 探索代码后出实施计划" |
| 审核 Plan | "第 3 点不需要改 Infrastructure，移除" |
| 开始实现 | "开始实现，从 Domain 层开始" |
| 每层完成 | "编译通过，继续 Application 层" |
| 验证 | "启动系统，按 Spec 验收标准逐条验证" |
| 提交 | "帮我提交，commit message 按我们的格式" |

### 8.2 避免的坑

| 坑 | 解法 |
|---|------|
| AI 一次改太多文件 | 明确说 "一次只改一层" |
| AI 顺便重构了无关代码 | 说 "不要动其他代码，只改目标文件" |
| AI 改了命名风格 | CLAUDE.md 第 3 条已约束 |
| AI 编造了不存在的 API | 说 "先读现有代码再改，不确定的问我" |
| Spec 写太模糊 | 按模板写，验收标准必须可执行 |

### 8.3 充分利用 AI 记忆

项目的 `memory/` 目录存储了架构、权限、模板系统的设计决策。AI 在每次会话开始时会读取这些记忆，**不需要每次都重新解释项目背景**。

新发现的重要决策 → 告诉 AI "记住这个" → AI 写入 `memory/` → 下次会话自动加载。

## 9. 速查卡

```
┌────────────────────────────────────────────────────────────┐
│                    变更执行速查卡                            │
├────────────────────────────────────────────────────────────┤
│  □ 写 Spec → docs/specs/YYYY-MM-DD-功能名.md                │
│    用模板，验收标准必须可验证                                  │
│                                                            │
│  □ AI 出 Plan → 审核文件清单 + 变更顺序                       │
│    不确定的让 AI 重新探索                                     │
│                                                            │
│  □ 分层实现 → 后端自底向上 / 前端自顶向下                      │
│    Domain → Application → Infrastructure → API → Frontend   │
│    每层: 改完 → 编译通过 → 下層                              │
│                                                            │
│  □ 验证 → 对照 Spec 验收标准逐条验证                           │
│    不通过 → 回去改 → 再验证                                   │
│                                                            │
│  □ 提交 → feat:/fix: + 描述 + 关联 Spec                      │
│    一个 Spec = 一个 commit                                   │
└────────────────────────────────────────────────────────────┘
```
