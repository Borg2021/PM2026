# Plan — 甘特图虚拟起止节点

> 日期：2026-06-29　|　状态：📋 待审批　|　关联 Spec：[20260629-甘特图虚拟起止节点.md](../specs/20260629-甘特图虚拟起止节点.md)

## 0. 目标

在甘特图中引入逻辑虚拟节点「Start」和「End」，使任务网络从多源多汇变为单源单汇，为核心功能**关键路径分析（CPM）** 提供标准算法的前提条件。

### Step 2 — `ganttFlattenedTasks` 插入虚拟行

**位置**：`ganttFlattenedTasks` 计算属性（L2616–2687）

**修改逻辑**：

```
1. 现有 walk + 二次处理不变
2. 在 result 生成后：
   a. 计算 startDate = 所有叶子任务中最小的 planStartDate（确保日期有效）
   b. 计算 endDate：
      - 收集所有被引用的任务 ID（遍历所有 preTaskCodes）
      - 叶子任务中 ID 不在被引用集合里 → 无后继
      - endDate = 无后继叶子中最晚的 planFinishDate
   c. 如果 result 非空：
      - unshift Start 虚拟行（barLeft=calcLeft(startDate), barWidth=0, index 重新编号）
      - push    End   虚拟行（barLeft=calcLeft(endDate),   barWidth=0, index 重新编号）
3. index 重新编号：遍历 result，令每个 item.index = 循环索引
```

**注意**：
- `barLeft` 计算需要 `timelineStart` 已存在（即 `ganttEarliestDate` 非空），若日期无效则跳过
- Start/End 的 `calcLeft` 与现有任务完全一致，保证位置同步
- 重新编号必须在 unshift/push 之后进行，确保 index 从 0 到 N+1 连续

**验证**：控制台打印 `ganttFlattenedTasks.value.length`，确认加了 2。

### Step 5 — `ganttDependencyLines` 补充 Start/End 连线

**位置**：`ganttDependencyLines` 计算属性（L2797–2823）

**当前逻辑**：遍历 flat，解析 `preTaskCodes`，画 FS 连线。

**新增逻辑**：

```
1. 构建被引用 ID 集合（现有 preTaskCodes 中出现的 taskId）
   const referencedIds = new Set<number>()
   for (const item of flat) {
     for (const code of parsePreTaskCodes(item.task.preTaskCodes)) {
       referencedIds.add(code.taskId)
     }
   }

2. Start → 无前置叶子：遍历 flat 中所有叶子任务
   若 preTaskCodes 为空（或 filtered 后为空）且 task.id > 0
   → 画线从 Start (barLeft=0, y=0*GANTT_ROW_H) 到 item

3. 无后继叶子 → End：遍历 flat 中所有叶子任务
   若 task.id 不在 referencedIds 中 且 task.id > 0
   → 画线从 item 到 End (barLeft=endBarLeft, y=(N+1)*GANTT_ROW_H)
```

**注意**：
- `barLeft` 使用 `ganttFlattenedTasks` 中的值，Start 和 End 的 barLeft 已在 Step 2 计算好
- Start 的 `barLeft + barWidth` = `barLeft`（宽度为 0）
- 连线只对**叶子任务**（`isLeaf`），父/汇总节点不参与

**验证**：无前置任务从 Start 收到连线，无后继任务指向 End。

### Step 6 — `criticalPathData` CPM 重构（核心）

**位置**：`criticalPathData` 计算属性（L2834–2955）

**当前问题**：
- 无前置任务 ES = 0，但实际项目有不同起始日期
- 无后继任务的 LF = projectEnd（各终点各自对齐），关键路径可能遗漏首尾段
- 多个零入度节点各自为"起点"

**重构方案**：

```ts
// 1. 把 Start 和 End 加入 taskMap（使其参与 CPM 计算）
const startItem = ganttFlattenedTasks.value.find(it => it.task.id === VIRTUAL_START_ID)
const endItem   = ganttFlattenedTasks.value.find(it => it.task.id === VIRTUAL_END_ID)
if (startItem && endItem) {
  const fakeStartTask = { ...startItem.task, id: VIRTUAL_START_ID }
  const fakeEndTask   = { ...endItem.task,   id: VIRTUAL_END_ID }
  // Map 中额外的 Start/End entry（仅用于 CPM，不影响渲染）
}

// 2. Start 入度 = 0（与现有逻辑一致），End 出度 = 0
//    Start → 所有无前置叶子（补入度边）
//    所有无后继叶子 → End（补出度边）
//    这些边加到 succMap / inDeg 中

// 3. 拓扑排序：Start 一定是 topo[0]（入度 = 0），End 一定是 topo 最后一个

// 4. 前向传递：
//    Start.ES = Start.EF = 0（相对最早日期的天数偏移）
//    对每个无前置叶子：ES = Start.EF + 0 = 0（统一起点）
//    对每个有前置任务：ES = max(前置.EF + lag)（现有逻辑不变）
//    End.ES = max(所有无后继.EF)，End.EF = End.ES + 0

// 5. 后向传递：
//    End.LF = End.ES（关键路径总长度）
//    对每个无后继叶子：LF = End.LS
//    对每个有后继任务：LF = min(后继.LS)（现有逻辑不变）
//    逆向传递至 Start

// 6. 总时差：
//    TS = LS - ES，TS = 0 → 关键任务
//    Start 和 End 分别在两端
```

**关键变化总结**：

| 项目 | 旧逻辑 | 新逻辑 |
|------|--------|--------|
| 无前置任务 ES | `0` | `Start.EF = 0`（通过 Start 边传播） |
| 无后继任务 LF | `projectEnd`（不统一） | `End.LS`（统一终点的后向约束） |
| 前向传递终点 | `max(所有.EF)` | `End.ES = max(所有无后继.EF)` |
| 后向传递起点 | `max(所有.EF)` | `End.LF = End.ES` |

**验证**：
- 控制台输出每个叶子任务的 ES/EF/LS/LF/float
- 关键路径上 Start→第一个关键任务 和 最后一个关键任务→End 被纳入
- 原来无前置的关键任务不会因为首段缺失而被遗漏

## 5. 风险点

| 风险 | 缓解 |
|------|------|
| `index` 重新编号影响依赖连线 y 坐标 | reindex 统一在 insert 后执行，`ganttDependencyLines` 直接读 `item.index` |
| 虚拟行 `barLeft` 计算依赖 `timelineStart` | 若 `ganttEarliestDate` 为空则跳过整个虚拟节点逻辑 |
| CPM 重构引入回归 | 保留现有 debug 日志输出，对比新旧 ES/EF/LS/LF 值 |
| 虚拟节点 index = 0 影响 `calcSeqAutoWidth` | 虚拟 taskNo 长度 ≤2，不影响实际宽度 |
