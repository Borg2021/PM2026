# 项目删除改为软删除

> 日期：2026-07-02　|　状态：📝 需求设计（待确认后实施）

## 1. 需求背景

### 1.1 现状

当前项目删除为**物理删除**（`_db.Projects.Remove(project)`），`ProjectCode` 字段设有数据库唯一索引。物理删除存在以下问题：

- 误删后数据无法恢复
- 项目下的关联数据（任务、文件、财务、变更记录、操作日志等）一并物理清除，审计追溯断裂
- 不满足企业级数据留存和合规要求

### 1.2 目标

将项目删除改为**软删除**，删除后项目数据完整保留，仅标记为已删除状态，同时通过重命名释放项目编号供后续复用。

## 2. 功能描述

### 2.1 软删除机制

| 行为 | 说明 |
|------|------|
| 标记删除 | 设置 `IsDeleted = true`，**不删除数据库行** |
| 编号重命名 | `ProjectCode` 追加 `_del_YYYYMMDD` 后缀（如 `808` → `808_del_20260702`） |
| 名称重命名 | `ProjectName` 追加 `_del_YYYYMMDD` 后缀（如 `某某项目` → `某某项目_del_20260702`） |
| 唯一约束 | 重命名后的编号天然唯一（含删除日期），不会与现有或未来的项目编号冲突 |
| 级联关系 | 子表数据（任务、成员、里程碑、文件、财务、变更记录、操作日志）**全部保留不动** |

### 2.2 实体变更

`Project` 实体新增字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| IsDeleted | bool | `false` | 软删除标记 |
| DeletedAt | DateTime? | `null` | 删除时间 |
| DeletedBy | long? | `null` | 删除人 Id |

### 2.3 删除操作流程

```
前端点击「删除」→ 确认弹窗 → 调用后端接口 →
  1. 权限校验（不变）
  2. 读取当前日期，格式化为 YYYYMMDD
  3. 更新 ProjectCode = 原值 + "_del_" + YYYYMMDD
  4. 更新 ProjectName = 原值 + "_del_" + YYYYMMDD
  5. 设置 IsDeleted = true, DeletedAt = DateTime.UtcNow, DeletedBy = 当前用户 Id
  6. 记录操作日志："删除项目「{原名称}」"
  7. 返回成功
```

### 2.4 查询过滤

所有项目查询（列表、详情、复制源校验等）默认排除已删除项目：

| 接口 | 过滤规则 |
|------|---------|
| 项目列表 `GET /projects` | `WHERE IsDeleted = false` |
| 项目详情 `GET /projects/{id}` | 软删除项目返回 404（`code: 404, "项目不存在或已删除"`） |
| 复制项目 `POST /projects/{id}/copy` | 源项目已删除 → 返回 404 |
| 状态变更 `PUT /projects/{id}/status` | 已删除项目拒绝操作 |
| 项目编号唯一性校验 | **仅校验未删除项目**（`WHERE IsDeleted = false AND ProjectCode = xxx`），已删除的编号不参与冲突检测 |

### 2.5 前端变更

| 改动点 | 说明 |
|--------|------|
| 删除确认文案 | 由「此操作不可恢复」改为「删除后项目将被标记为已删除，项目编号和名称将追加删除标记，可联系管理员处理」 |
| 项目列表 | 不显示已删除项目（后端过滤） |
| 已删除项目详情 | 如通过直接 URL 访问，后端返回 404，前端显示「项目不存在或已删除」 |

## 3. 权限设计

不新增权限码，沿用现有 `project:delete` 权限。删除权限的角色分配不变：

| 角色 | 可删除 |
|------|:--:|
| 系统管理员 | ✓ |
| 项目管理员 | ✓ |
| 项目经理 | — |

> 维持现有逻辑：`ForbidIfCannotMaintainAsync` 校验维护权限。

## 4. API 设计

### 4.1 删除接口

**不新增接口**，仅改造现有 `DELETE /api/v1/projects/{id}` 的行为：

- 请求/响应格式不变
- 内部由物理删除改为软删除 + 重命名

### 4.2 查询接口变更

所有涉及 `Project` 的查询在 Repository 层统一追加 `IsDeleted == false` 过滤：

| 接口 | 改动 |
|------|------|
| `GET /projects` | Repository `GetListAsync` 增加 `.Where(p => !p.IsDeleted)` |
| `GET /projects/{id}` | `GetByIdAsync` 内部增加 `IsDeleted` 判断，已删除返回 null |
| `POST /projects/{id}/copy` | 源项目查询走 `GetByIdAsync`，自然被过滤 |

### 4.3 项目编号唯一性校验变更

```diff
- await _repo.ProjectCodeExistsAsync(code)
+ await _repo.ProjectCodeExistsAsync(code)  // 内部增加 IsDeleted == false 过滤
```

## 5. 数据库变更

### 5.1 新增列

```sql
ALTER TABLE Projects ADD IsDeleted BIT NOT NULL DEFAULT 0;
ALTER TABLE Projects ADD DeletedAt DATETIME2 NULL;
ALTER TABLE Projects ADD DeletedBy BIGINT NULL;
```

### 5.2 索引调整

现有 `ProjectCode` 唯一索引保持不变。软删除后项目编号含删除日期后缀，天然不冲突。

### 5.3 初始化

`DbInitializer.EnsureSeedAsync` 中自动执行列补充（沿用现有自愈机制：检测缺失列 → ALTER TABLE ADD）。

## 7. 边界与约束

| 场景 | 处理方式 |
|------|---------|
| 重复删除 | 已删除项目再次调用删除接口 → 返回 404 "项目不存在或已删除" |
| 同名项目编号 | 删除 `808` 后新建 `808` → 允许（唯一性校验仅查未删除项目） |
| 同一天多次删除同一项目 | 不会发生（已删除后接口返回 404） |
| 删除后复制 | 复制源为已删除项目 → 返回 404 |
| 删除后编辑 | 已删除项目拒绝更新 → 返回 404 |
| 删除后状态变更 | 已删除项目拒绝状态操作 → 返回 404 |
| 已删除项目的数据导出 | 本期不做，后续按需增加 |
| 恢复已删除项目 | 本期不做，由管理员通过数据库直接操作 |

## 9. 设计决策记录

| 决策点 | 结论 | 原因 |
|--------|------|------|
| 删除方式 | 软删除（标记 + 重命名） | 保护数据完整性，释放编号供复用 |
| 编号后缀格式 | `_del_YYYYMMDD` | 简洁、可排序、含时间信息、保证唯一 |
| 名称同步重命名 | 是 | 列表展示时一眼可辨已删除 |
| 过滤策略 | 所有查询默认排除 | 已删除项目不应出现在正常业务流程中 |
| 恢复功能 | 本期不做 | 需求未涉及，避免范围蔓延 |
| 唯一索引 | 保持不变 | 后缀天然保证唯一，无需改索引 |
| 子表数据 | 全部保留 | 审计追溯需要 |
