# 任务计划工期计算偏差分析

> 日期：2026-06-29　|　状态：已实现

## 1. 现象

**项目**：909 玖物阿普塔二期  
**节点**：001  
**数据**：

| 字段 | 值 |
|------|-----|
| 计划开始 | 2026-05-30 |
| 计划完成 | 2026-11-02 |
| **计划工期（显示）** | **1209 天** |
| 实际日历跨度 | ≈ 156 天（2026-11-02 - 2026-05-30） |

用户期望计划工期 = 156 天，但系统显示 1209 天，偏差高达 **1053 天**。

## 2. 根因分析

### 2.1 核心问题：父节点工期计算逻辑错误

追溯代码 `ProjectEditor.vue:1269-1274`（`syncParentPlanDates` 函数）：

```javascript
// 计划工期 = 所有直接子节点计划工期之和
const totalDuration = children.reduce((sum, c) => sum + (c.planDuration || 0), 0)
if (parent.planDuration !== totalDuration) {
  parent.planDuration = totalDuration
}
```

**父节点的 `planDuration` 是子节点工期之和，而非父节点自身日期跨度（planFinishDate - planStartDate）。**

### 2.2 偏差如何产生

对于一个有多个子节点的父节点：

| 数据 | 计算方式 | 本例 |
|------|---------|------|
| `planStartDate` | = `min(子节点.planStartDate)` | 2026-05-30 ✅ |
| `planFinishDate` | = `max(子节点.planFinishDate)` | 2026-11-02 ✅ |
| `planDuration` | = **Σ(子节点.planDuration)** ← 错误 | **1209** ❌ |

由于子任务之间存在**并行关系**（多个子任务在同一时间段内并行执行），将各子任务的工期直接相加会把并行的工期叠加，导致父节点工期被严重高估。

**示例说明**：

```
父节点 001 (2026-05-30 ~ 2026-11-02, 工期=?）
├── 子 A: 2026-05-30 ~ 2026-08-15  工期=77天
├── 子 B: 2026-06-01 ~ 2026-09-30  工期=121天
├── 子 C: 2026-07-01 ~ 2026-11-02  工期=124天
├── 子 D: ............              工期=XX天
├── ... (更多并行子任务)
└── 子 N: ............              工期=XX天

父节点日历跨度 = 2026-11-02 - 2026-05-30 = 156 天
父节点工期(当前) = 77 + 121 + 124 + ... = 1209 天  ← 语义错误
父节点工期(期望) = 156 天                         ← 日历跨度
```

### 2.3 语义混淆

| 概念 | 含义 | 适用场景 |
|------|------|---------|
| **计划工期** | 工作包的日历跨度 = `完成日期 - 开始日期` | 甘特图横条宽度、进度计算 |
| **累计工期** | 所有子任务工期之和（含并行） | 人天估算、资源负荷 |

当前代码将「累计工期」赋值给了「计划工期」字段，造成了数值异常。

### 2.4 影响链路

`syncParentPlanDates()` 在以下时机被调用（共 **5 处**）：

| 行号 | 调用场景 |
|------|---------|
| 1740 | 新增任务后保存 |
| 1776 | `saveListEdits()` 内（行内编辑保存前） |
| 2075 | `syncParentAndCollectChangedNos()` 内（级联传播前置任务约束） |
| 2250 | `resetPlanDates()` 内 |
| 2295 | `loadTasks()` 加载任务数据后 |

每次触发都会重新将父节点 `planDuration` 覆写为子节点之和，偏差会持续存在且可能在级联传播中被放大。

## 3. 修正方案（已实施）

父节点的计划工期应改为：

```
父节点.planDuration = planFinishDate - planStartDate（日历天数）
```

即删除子节点工期求和逻辑，改为从父节点自身的起止日期计算日历跨度。

**涉及的连锁影响需评估**：

- **进度加权平均**（line 1281）：`weightedPct = Σ(child.planDuration × child.progressPct) / Σ(child.planDuration)`，此处使用的是子节点的 `planDuration`，不受父节点工期计算方式影响 ✅
- **甘特图渲染**：使用 `planStartDate` / `planFinishDate` 绘制横条，不直接依赖 `planDuration` ✅
- **实际工期**（line 1276）：`actualDuration` 同样使用了求和逻辑，需一并修正
- **前端导出 Excel**（line 1880）：导出的「计划工期」列会同步变化

## 5. 修正内容

> 日期：2026-06-29

将 `syncParentPlanDates()` 中父节点的计划工期和实际工期从「子节点工期之和」改为「父节点自身日期跨度」：

```javascript
// 修正前（错误）
parent.planDuration = Σ child.planDuration    // 子节点工期累加，并行时段重复计算
parent.actualDuration = Σ child.actualDuration

// 修正后（正确）
parent.planDuration = planFinishDate - planStartDate      // 日历天数
parent.actualDuration = actualFinishDate - actualStartDate // 日历天数
```

进度加权平均计算不变，仍使用子节点的 `planDuration` 做加权分母。
