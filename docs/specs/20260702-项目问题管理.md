# 项目问题管理

> 日期：2026-07-02　|　状态：📝 需求设计（待确认后实施）

## 1. 需求背景

当前系统覆盖了项目台账、WBS 任务、文件管控、财务管理、变更管理五大业务域，但在**问题追踪**方面仍是空白。项目执行过程中发现的各类问题（技术缺陷、进度风险、资源问题等）缺少结构化的记录、指派、跟踪和关闭机制，目前依赖线下沟通或变更管理模块临时承载，既不规范也难以量化考核。

需要在现有项目体系上新增「问题管理」模块，为每个项目提供独立的问题台账，覆盖问题从发现到关闭的完整生命周期。

### 1.1 核心目标

- 项目下统一管理各类问题，通过字典区分类型和来源
- 问题 → 措施（1:N），每条措施可指派责任部门和责任人
- 简洁三态流转（待处理 → 处理中 → 已完成），重新打开需记录次数作为考核依据
- 复用现有权限体系（数据范围）+ 新增独立操作权限码
- 附件复用现有项目文件体系
- 侧边栏「项目管理」目录下新增全局「问题管理」入口，跨项目汇总当前用户相关问题
- 项目问题与项目状态（激活/完成/暂停）解耦，任何状态的项目均可创建和管理问题

## 2. 功能描述

### 2.1 问题主表

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| Id | long | — | 主键，自增 |
| ProjectId | long | ✓ | 所属项目 |
| IssueCode | string | ✓ | 问题编号，格式 `{ProjectCode}-ISS-{序号}`（如 `PRJ001-ISS-001`），项目内自增 |
| Title | string | ✓ | 问题标题 |
| Description | string | — | 问题描述（富文本） |
| IssueSource | string | ✓ | 问题来源（字典 `issue_source`） |
| IssueType | string | ✓ | 问题类型（字典 `issue_type`） |
| Severity | string | ✓ | 严重程度：`重要` / `一般` |
| Priority | string | ✓ | 优先级：`紧急` / `一般` |
| Status | int | ✓ | 0 = 待处理 / 1 = 处理中 / 2 = 已完成 |
| CauseAnalysis | string | — | 原因分析 |
| DiscoveredDate | DateOnly | ✓ | 发现日期 |
| PlannedDate | DateOnly | — | 计划完成 |
| ResponsibleDeptId | long | — | 责任部门，选择责任人时自动带出，也可先选部门再选人 |
| AssigneeId | long | ✓ | 责任人（关联 User），必填 |
| SubmitterId | long | ✓ | 提出人（关联 User），手动选择 |
| CreatorId | long | ✓ | 录入人，自动取当前登录用户 |
| VerifierId | long | — | 验证人（关联 User），选填 |
| VerifiedDate | DateOnly? | — | 验证日期，选填 |
| ReopenCount | int | — | 重新打开次数，默认 0 |
| CreatedAt | DateTime | — | 创建时间 |
| UpdatedAt | DateTime | — | 更新时间 |

### 2.2 措施子表 `ProjectIssueMeasure`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| Id | long | — | 主键，自增 |
| IssueId | long | ✓ | 关联问题，级联删除 |
| SortOrder | int | ✓ | 序号，前端拖拽排序 |
| Measure | string | ✓ | 措施内容 |
| MeasureType | string | ✓ | 措施类型（字典 `measure_type`） |
| ResponsibleDeptId | long | — | 责任部门 |
| ResponsibleUserId | long | — | 责任人 |
| Remark | string | — | 备注 |
| PlannedDate | DateOnly | — | 计划完成 |

### 2.3 附件

不新建附件表，问题附件直接存入现有 `ProjectFileItem` + `ProjectFileVersion` 体系。通过 `ProjectFileItem` 的 `SourceType` 字段区分来源（新增 `"issue"` 枚举值），`SourceId` 关联问题 Id。

文件可见性默认跟随项目权限，与现有文件管控保持一致。

### 2.4 状态流转

```
待处理(0) ──→ 处理中(1) ──→ 已完成(2)
                 ↑              │
                 └──────────────┘
                  重新打开 (ReopenCount++)
```

| 操作 | 当前状态 | 目标状态 | 权限要求 |
|------|---------|---------|---------|
| 开始处理 | 待处理(0) | 处理中(1) | `issue:edit` |
| 完成 | 处理中(1) | 已完成(2) | `issue:edit` |
| 重新打开 | 已完成(2) | 处理中(1) | `issue:edit`，`ReopenCount += 1` |

### 2.5 问题编号规则

- 格式：`{项目编号}-ISS-{序号}`
- 序号为项目内自增（3 位补零，如 001、002）
- 创建时自动生成，不可编辑
- 删除问题后序号不回收

### 2.6 全局「问题管理」菜单（跨项目个人视图）

在侧边栏「项目管理」目录下，「文件管理」之前，新增「问题管理」菜单项。该页面是一个**跨项目视图**，汇总当前登录用户相关的所有问题，方便用户在一个页面集中处理与自己有关的 issue。

**菜单位置**（侧边栏）：

```
项目管理
  ├── 项目管理        ← 现有：项目列表
  ├── 任务管理        ← 现有：跨项目任务视图
  ├── 问题管理        ← 新增：跨项目问题视图
  └── 文件管理        ← 现有：跨项目文件视图
```

**页面功能**：

| 区域 | 说明 |
|------|------|
| 筛选区 | 除 4.2 节公共筛选外，额外支持**所属项目**下拉筛选 |
| 表格列 | 与项目内问题列表一致，**额外增加「所属项目」列**（显示项目编号+名称，点击跳转项目详情） |
| 数据范围 | 仅显示当前用户相关的项目问题（责任人是自己 OR 提出人是自己 OR 录入人是自己 OR 验证人是自己），叠加现有数据范围规则 |
| 新建问题 | 支持在页面内直接创建问题，需先选择**所属项目**（下拉框），其余字段与项目内创建一致 |

**数据过滤逻辑**（后端）：

```
WHERE (AssigneeId = currentUserId 
   OR SubmitterId = currentUserId 
   OR CreatorId = currentUserId 
   OR VerifierId = currentUserId)
  AND 项目数据范围过滤（沿用现有 Project 行级权限）
```

**待办角标**：侧边栏「问题管理」菜单旁显示当前用户待处理问题数（`Status != 2` 且满足以上关联条件），参考「我的工作台」待办任务计数实现。

### 2.7 问题与项目状态解耦

项目问题的生命周期**完全独立于**项目状态。无论项目处于什么状态（未激活/进行中/已完成/已暂停），用户都可以：

| 项目状态 | 创建问题 | 编辑问题 | 状态流转 | 删除问题 |
|---------|:--:|:--:|:--:|:--:|
| 未激活(0) | ✓ | ✓ | ✓ | ✓ |
| 进行中(1) | ✓ | ✓ | ✓ | ✓ |
| 已完成(2) | ✓ | ✓ | ✓ | ✓ |
| 已暂停(3) | ✓ | ✓ | ✓ | ✓ |

> 设计原因：已完成项目的遗留问题仍需跟踪关闭，已暂停项目的问题仍需记录和推进，问题管理不应受项目生命周期约束。

## 3. 权限设计

### 3.1 数据范围（复用现有逻辑）

问题列表的数据可见范围沿用现有项目数据范围规则，逐角色定义：

| 角色 | 可见范围 |
|------|---------|
| 系统管理员 | 全部项目的问题 |
| 项目管理员 | 本部门及子级项目的问题 |
| 项目经理 | 自己名下项目的问题 |
| 任务管理 | 本人为项目成员的项目的问题 |
| 普通用户 | 本人为项目成员的项目的问题 |

### 3.2 操作权限（新增权限码）

在权限树 `project` 节点下新增 `issue` 分支：

```
project
  ├── issue                    ← 菜单入口
  │   ├── issue:list           ← 查看问题列表
  │   ├── issue:create         ← 创建问题
  │   ├── issue:edit           ← 编辑问题（含状态变更、重新打开）
  │   ├── issue:delete         ← 删除问题
  │   ├── issue:assign         ← 指派责任人
  │   └── issue:verify         ← 验证问题（填写验证人、验证日期）
```

### 3.3 默认角色分配

| 权限码 | 系统管理员 | 项目管理员 | 项目经理 | 任务管理 | 普通用户 |
|--------|:--:|:--:|:--:|:--:|:--:|
| `issue:list` | ✓ | ✓ | ✓ | ✓ | ✓ |
| `issue:create` | ✓ | ✓ | ✓ | ✓ | — |
| `issue:edit` | ✓ | ✓ | ✓ | — | — |
| `issue:delete` | ✓ | ✓ | — | — | — |
| `issue:assign` | ✓ | ✓ | ✓ | — | — |
| `issue:verify` | ✓ | ✓ | ✓ | — | — |

> Admin 角色自动豁免所有权限检查，继承现有 `[RequirePermission]` 机制。

## 4. API 设计

### 4.1 路由总览

**项目级接口**（挂在项目路由下）：

| 方法 | 路由 | 说明 |
|------|------|------|
| `GET` | `/api/v1/projects/{projectId}/issues` | 项目内问题列表（分页 + 筛选） |
| `GET` | `/api/v1/projects/{projectId}/issues/{id}` | 问题详情（含措施列表 + 附件） |
| `POST` | `/api/v1/projects/{projectId}/issues` | 创建问题（含措施） |
| `PUT` | `/api/v1/projects/{projectId}/issues/{id}` | 编辑问题（含措施整批替换） |
| `DELETE` | `/api/v1/projects/{projectId}/issues/{id}` | 删除问题（级联删除措施） |
| `PUT` | `/api/v1/projects/{projectId}/issues/{id}/status` | 变更问题状态 |

**全局接口**（跨项目个人视图）：

| 方法 | 路由 | 说明 |
|------|------|------|
| `GET` | `/api/v1/issues/my` | 当前用户相关问题列表（分页 + 筛选） |
| `GET` | `/api/v1/issues/my/count` | 当前用户待处理问题计数（用于侧边栏角标） |
| `POST` | `/api/v1/issues` | 创建问题（需传 `projectId`，不依赖项目状态） | |

### 4.2 列表筛选参数

**项目内问题列表**（`/api/v1/projects/{projectId}/issues`）：

| 参数 | 类型 | 说明 |
|------|------|------|
| pageIndex / pageSize | int | 分页 |
| keyword | string | 标题 / 编号模糊搜索 |
| issueType | string | 问题类型 |
| issueSource | string | 问题来源 |
| severity | string | 严重程度 |
| priority | string | 优先级 |
| status | int | 状态 |
| assigneeId | long | 责任人 |
| submitterId | long | 提出人 |
| responsibleDeptId | long | 责任部门 |
| startDate / endDate | DateOnly | 发现日期范围 |

**全局问题列表**（`/api/v1/issues/my`）：在以上筛选基础上，额外增加：

| 参数 | 类型 | 说明 |
|------|------|------|
| projectId | long | 所属项目筛选（可选） |

### 4.3 状态变更请求体

```json
{
  "status": 1,
  "causeAnalysis": "原因分析内容（可选）"
}
```

- 重新打开时 `status=1`，后端自动递增 `ReopenCount`
- 完成时 `status=2`，可同时填写验证人和验证日期

### 4.4 统一响应

沿用现有 `ApiResponse<T>` 封装：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "success": true
}
```

## 5. 前端设计

### 5.1 入口一：项目详情页 Tab

项目详情页新增 Tab「问题管理」，与其他 Tab（任务计划/里程碑/考核任务/文件资料/变更管理/财务管理）并列。

### 5.2 入口二：侧边栏全局菜单

侧边栏「项目管理」目录下，「文件管理」之前新增「问题管理」菜单项，路由 `/project/issues`。点击进入跨项目个人问题视图（见 2.6 节）。

### 5.3 项目内问题列表

**表格列**：编号、标题、类型、严重程度、优先级、状态、责任人、责任部门、发现日期、计划完成、重新打开次数、操作

**筛选区**：标题/编号关键字、类型、来源、严重程度、优先级、状态下拉

**状态 Tag 颜色**：
- 待处理 → 橙色 `warning`
- 处理中 → 蓝色 `primary`
- 已完成 → 绿色 `success`

**操作列按钮**（按当前状态显示）：
- 待处理 → 「开始处理」
- 处理中 → 「完成」
- 已完成 → 「重新打开」（需 `issue:edit` 权限）
- 所有状态 → 「编辑」「删除」（按权限控制）

### 5.4 全局问题列表（`/project/issues`）

参考「任务管理」（`/project/tasks`）的实现模式，提供跨项目的问题汇总视图。

**表格列**：在项目内列表基础上，在最左侧增加「所属项目」列（显示 `项目编号 - 项目名称`），点击可跳转项目详情。

**筛选区**：在项目内筛选基础上，顶部额外增加「所属项目」下拉框（支持搜索）。

**数据范围**：仅显示当前用户相关的问题（责任人是自己 OR 提出人是自己 OR 录入人是自己 OR 验证人是自己），附加现有项目数据范围。

**创建问题**：页面顶部「新建问题」按钮，点击弹出创建对话框，**比项目内创建多一个「所属项目」下拉框**（必选，用于指定问题归属的项目）。

**角标**：侧边栏「问题管理」菜单显示待处理问题数角标（`Status != 2 && 用户相关`），参考现有「我的工作台」待办计数实现。

**状态操作**：支持行内状态变更（开始处理/完成/重新打开），与项目内问题列表一致。

### 5.5 创建/编辑弹窗

弹窗分三个区域以手风琴或分区卡片展示：

**区域一：问题基本信息**
- 表单字段按主表字段排列
- 责任部门选择后自动过滤部门下的人员供责任人选择
- 责任人选择后自动回填责任部门
- 计划完成默认为空

**区域二：措施清单**
- 动态行表格，支持添加/删除/拖拽排序
- 每行包含：序号、措施内容、措施类型、责任部门、责任人、备注、计划完成
- 至少保留一行空白行

**区域三：附件**
- 复用现有文件上传组件
- 创建时暂不上传（先保存问题生成 Id），编辑时完整支持

### 5.6 详情页

点击编号或标题可展开详情抽屉（或弹窗），展示：
- 完整基本信息
- 措施清单（只读表格）
- 附件列表（可下载）
- 操作日志（创建、编辑、状态变更、重新打开）

## 8. 边界与约束

| 场景 | 处理方式 |
|------|---------|
| 项目无成员 | 责任人/提出人下拉为空，提示先添加项目成员 |
| 删除项目 | 级联删除项目下所有问题及措施 |
| 责任人被移除出项目 | 已分配的问题保留（历史记录），不作为数据一致性校验 |
| 措施清单为空 | 允许保存（问题可只有标题，措施后续补充） |
| 重新打开次数上限 | 无上限，持续累加 |
| 问题编号前缀 | 通过系统参数 `issue_code_prefix` 动态读取，更改后新问题生效 |
| 已完成/已暂停项目 | 问题管理功能不受项目状态影响，可正常创建、编辑、流转 |
| 全局视图新建问题 | 需选择所属项目（下拉框），项目列表不含数据范围外的项目 |
| 全局视图数据过滤 | 仅显示当前用户相关的问题（责任/提出/录入/验证四个角色之一），叠加项目数据范围 |
| 角标计数 | 仅统计 `Status != 2` 的问题数，与项目状态无关 |

## 10. 设计决策记录

| 决策点 | 结论 | 原因 |
|--------|------|------|
| 问题关联粒度 | 仅关联到项目 | 保持简洁，避免过度耦合任务/里程碑 |
| 状态流转 | 三态（待处理/处理中/已完成） | 足够覆盖核心场景，复杂度可控 |
| 重新打开 | 记录次数不设上限 | 作为考核依据，需完整记录 |
| 附件方案 | 复用现有文件体系 | 不重复造轮子，统一文件管理 |
| 问题类型 | 字典可配 | 灵活，符合系统现有模式 |
| 措施表 | 1:N 子表 | 用户明确要求多条措施 |
| 提出人 | 手动选择 | 提出人可能不是系统操作者 |
| 录入人 | 自动取当前用户 | 审计需要，与提出人分离 |
| 全局问题视图 | 侧边栏独立菜单项 | 方便用户集中处理与自己相关的问题，参考「我的任务」设计模式 |
| 问题与项目状态解耦 | 不受项目状态限制 | 已完成项目的遗留问题仍需跟踪，已暂停项目问题仍需推进 |
| 菜单位置 | 「项目管理」目录「文件管理」之前 | 用户明确指定位置 |
