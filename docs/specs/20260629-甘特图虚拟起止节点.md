# [甘特图增加虚拟「开始」与「结束」节点]

> 日期：2026-06-29　|　状态：📝 需求设计（待确认后实施）

## 1. 需求背景

当前甘特图中，无前置任务的任务在时间轴上**孤立显示**，缺少统一的起点约束；无后继任务的任务**无明显终点标识**。这在项目整体时间分析中造成视觉分散，不利于快速判断项目总工期和空闲时差。

引入逻辑上的（非实际数据库实体）「开始」和「结束」两个虚拟节点，为甘特图提供**单一入口**和**单一出口**，使其具备标准 PERT/CPM 网络图的语义基础。

### 1.1 核心用途：关键路径显示

虚拟起止节点**首要服务于关键路径分析**。当前 CPM 计算存在结构性问题：

```
现状（无虚拟节点）：
  多个无前置任务各自成为"起点" → 前向传递需要从多个零入度节点同时起步
  多个无后继任务各自成为"终点" → 后向传递终点不唯一，总时差基准不统一

引入 Start/End 后：
  Start（单一起点）→ [任务网络] → End（单一终点）
  
  前向传递：Start.ES = Start.EF = 项目最早日期
            所有无前置任务从 Start 获得 ES，逐级正向传播
  后向传递：End.LS = End.LF = 项目最晚日期（关键路径长度）
            所有无后继任务向 End 对齐，逐级反向传播
  总时差：TS = LS - ES = LF - EF，TS = 0 即为关键任务
```

**没有 Start/End 时**，多个无前置任务的 ES 都等于各自最早日期（不统一），多个无后继任务的总时差计算基准不一致，导致关键路径可能**遗漏**首尾段。

**有 Start/End 后**，关键路径定义为一个闭合的数学模型：**从 Start 到 End 的最长路径**，路径上所有任务总时差 = 0。这是标准 CPM 算法的前提条件。

## 2. 功能描述

### 2.1 虚拟「开始」节点（Start）

| 属性 | 说明 |
|------|------|
| **性质** | 纯逻辑节点，**不**写入数据库 `ProjectTasks` 表 |
| **日期** | 计划开始 = 计划完成 = **所有任务节点中最小的 `planStartDate`** |
| **前置任务** | 无 |
| **后置任务** | 所有 `preTaskCodes` 为空（或不含有效 FS/SS 依赖）的叶子任务 |
| **甘特图表示** | 一条竖线（带 ◆ 标记），位于时间轴最早日期处 |

### 2.2 虚拟「结束」节点（End）

| 属性 | 说明 |
|------|------|
| **性质** | 纯逻辑节点，**不**写入数据库 `ProjectTasks` 表 |
| **日期** | 计划开始 = 计划完成 = **所有无后继任务的叶子节点中最晚的 `planFinishDate`** |
| **前置任务** | 所有不充当任何其他任务前置的叶子任务（即"出度"为 0 的任务） |
| **后置任务** | 无 |
| **甘特图表示** | 一条竖线（带 ◆ 标记），位于时间轴最晚日期处 |

### 2.3 关键定义

| 术语 | 定义 |
|------|------|
| **叶子任务** | 无 `children`（不包含子节点）的任务，即 `nodeType === 1` 且 `!taskHasChildren(id)` |
| **无后继任务** | 当前任务 ID 不出现在任何其他任务的 `preTaskCodes` 中 |
| **无前置任务** | 当前任务 `preTaskCodes` 为空字符串或只包含无效/已删除的引用 |
| **入度** | 依赖当前任务的前置任务数量（有多少任务指向它） |
| **出度** | 当前任务作为前置任务被多少任务引用 |

## 3. 甘特图视觉变化

### 3.1 时间轴

- 甘特图时间轴范围由 `ganttEarliestDate` / `ganttLatestDate` 决定
- **不需要改动**：最早的 `planStartDate` 和最晚的 `planFinishDate` 天然覆盖起止节点

### 3.2 虚拟节点渲染

```
        ◆ Start                 ◆ End
        │                       │
──┬─────┼───────────────────────┼─────┬──
  │     │    [任务条...]         │     │
  │     │                       │     │
```

- 两个节点各渲染为一个**菱形标记**（与里程碑样式一致，但颜色区分）
- Start 位于时间轴最早日期处（左侧）
- End 位于时间轴最晚日期处（右侧）
- 用虚线竖线贯穿整列

### 3.3 依赖连线

- **Start → 无前置任务的任务**：紫色依赖线（与其他 FS 依赖一致），从 Start 出发
- **无后继任务的任务 → End**：紫色依赖线，指向 End
- 连线使用现有 `buildDepPath` 算法，方向从 Start 到其直接后继、从无后继叶子到 End

## 5. 实现方案概述

### 5.1 `ganttFlattenedTasks` 扩展

当前结构：
```ts
interface GanttFlatItem {
  task: ProjectTaskItem; level: number; index: number
  barLeft: number; barWidth: number
  actualBarLeft: number; actualBarWidth: number
  displayStatus: number; actualDisplayStatus: number
}
```

扩展方案：在 `ganttFlattenedTasks` 计算中，**在数组头部 unshift** Start 虚拟行、**在尾部 push** End 虚拟行。

虚拟行的 `GanttFlatItem` 中：
- `task`：使用一个带特殊标记的虚拟对象（如 `id = -1`、`id = -2`，`nodeType = 2` 里程碑类型）
- `barLeft`：Start = 0（时间轴起点），End = 最晚 `planFinishDate` 对应位置
- `barWidth`：0（菱形标记无宽度）

### 5.2 `ganttDependencyLines` 扩展

```
为每个叶子任务 item 判断：
  1. preTaskCodes 为空 → 认为前置是 Start（添加入度连线 Start→item）
  2. 当前任务 ID 不在任何其他任务的 preTaskCodes 中 → 认为后继是 End（添加出度连线 item→End）
```

注意：这里的"入度/出度"判断需要遍历 `ganttFlattenedTasks` 中的所有 `preTaskCodes`，收集被引用任务的 ID 集合。

### 5.3 模板渲染

- 菱形标记：复用现有 `.gantt-milestone` CSS，但用不同颜色区分（如 Start 浅绿、End 浅红）
- 虚线列：用 `border-left: 1px dashed` 在虚拟节点列位置渲染

### 5.4 CPM 关键路径（核心场景）

Start 和 End 使项目任务网络从**多源多汇**变为**单源单汇**，CPM 算法成为标准的最长路径问题：

```
前向传递（Forward Pass）：

  1. Start.EF = Start.ES = 项目最早 planStartDate
  2. 对每个有前置任务的任务 T：
     T.ES = max(所有前置任务.EF + lag)   // 现有逻辑
     对于无前置任务的任务：
     T.ES = Start.EF                     // Start 作为唯一前置
     T.EF = T.ES + 工期
  3. 遍历完成后：
     End.ES = max(所有无后继任务.EF)     // 所有终点的最晚值
     End.EF = End.ES + 0                 // End 自身工期为 0

后向传递（Backward Pass）：

  1. End.LF = End.ES  (= 关键路径总长度)
     End.LS = End.LF
  2. 对每个任务 T（逆序）：
     T.LF = min(所有后继任务.LS)         // 现有逻辑
     无后继任务：T.LF = End.LS           // End 作为唯一后继
     T.LS = T.LF - 工期
  3. 向前回溯至 Start，Start.LS = Start.LF = 0

关键任务判定：
  TS（总时差）= LS - ES = LF - EF
  TS = 0 → 关键任务 → 连线标记为关键路径（黑色加粗虚线）
```

**为什么 Start/End 是关键路径的基础**：
- 无 Start 时：多个零入度任务各自 ES 不同，前向传递无统一基准
- 无 End 时：多个零出度任务各自 LF 不同，后向传递终点不唯一
- 有 Start/End 后：整个网络是闭合系统，最长路径计算有唯一起点和终点
- 关键路径上的 Start→首个任务 和 最后任务→End 段同样是关键路径的组成部分

## 6. 边界情况

| 场景 | 处理 |
|------|------|
| 项目无任务 | 不渲染 Start/End（`ganttFlattenedTasks` 为空 → 跳过） |
| 只有 1 个任务（有前置） | Start 和 End 可以指向同一个任务 |
| 只有 1 个任务（无前置） | Start→该任务→End 构成完整链条 |
| 前置任务引用已被删除的任务 ID | 不参与连线计算（现有逻辑已处理） |
| 任务的 planStartDate/planFinishDate 为空 | 不参与 Start/End 日期计算，跳过该任务的连线 |
| 任务为父节点（有 children） | 不参与入度/出度计算（只考虑叶子任务） |

## 8. 验证要点

1. 甘特图左侧出现 Start 菱形标记，位置在所有任务的最早计划开始日期处
2. 甘特图右侧出现 End 菱形标记，位置在所有无后继任务的最晚计划完成日期处
3. 无前置任务的叶子任务，其依赖连线从 Start 出发
4. 无后继任务的叶子任务（不出现在任何 preTaskCodes 中），其依赖连线指向 End
5. 中间任务（既有前置又有后继）的连线不受影响
6. 项目无任务时不显示虚拟节点
7. 虚拟节点不出现在任务计划列表（树表格）中，仅在甘特图可见
8. 切换日/周视图时虚拟节点位置正确跟随时间轴缩放
9. 虚拟节点对应的甘特图行高度与普通行一致，序号列为空或特殊标记
10. 关键路径分析中 Start 是天然起点（EF = 最早日期）

## 9. 前置条件

- `ganttFlattenedTasks` 中的 `index` 字段需要感知虚拟行（虚拟行 index = 0 和 index = N+1）
