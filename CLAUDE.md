# CLAUDE.md — Project Rules for AI（2026 成熟版）
## 🔒 核心铁律（必须严格遵守）
### 1. 先思考，再写代码（禁止瞎猜）
- 动手前先理解：需求、现有代码、依赖、接口、目录结构。
- 不确定的地方：**先问，不要假设，不要编造**。
- 不允许：凭空造参数、接口、配置、类型。

### 2. 极简主义（禁止过度设计）
- 只做需求内的事，**不加额外功能、不做未来扩展、不写冗余抽象**。
- 函数≤20行、文件≤200行；长了就拆，不要堆。
- 拒绝：接口+实现类+工厂这种“重型脚手架”，能用函数就不用类。

### 3. 外科手术式改动（禁止乱改无关代码）
- **只改必须改的行**，diff 越小越好。
- 不自动格式化、不统一引号、不重排 import、不顺便重构。
- 不碰：注释、空行、命名风格（除非明确要求）。

### 4. 目标驱动（不要步骤，要结果）
- 先写测试（验证目标），再写代码让测试通过。
- 不说“怎么做”，说“要达到什么结果”。
- 不做：无测试的代码、无目的的重构。

### 5. 先读再写（杜绝重复造轮子）
- 新增代码前，**先读同文件、依赖文件、公共工具**。
- 检查：是否已有可用函数/方法，不要重复实现。
- 必须注明：“已检查 utils/xxx.ts，xxx 可复用，无需新写”。

### 6. 约定优于创新（跟着现有风格走）
- 严格遵守项目现有：命名、格式、注释、错误处理、库版本。
- 即使你知道更好的方案，**也不要擅自替换**，先提建议。
- 禁止：混用风格（比如 JS 里突然用 Python 风格）。

### 7. 禁止大范围重构（必须最小变更）
- 一次任务只改一个文件/一个模块，**不跨目录批量修改**。
- 重构必须单独任务、单独 PR，不混在功能里。
- 不允许：未经确认就“优化结构”“统一架构”。

### 8. 安全与性能底线（不写垃圾代码）
- 禁止：硬编码密钥、明文密码、SQL 拼接、无校验输入。
- 禁止：any 类型（用 unknown）、空 catch、吞异常。
- 性能：不写 O(n²) 循环、不重复请求、不渲染冗余节点。

### 9. 测试要验证意图（不是凑覆盖率）
- 测试核心行为，不是实现细节。
- 断言必须**能失败**，不要写永远 true 的判断。
- 禁止：mock 一切、跳过关键用例、写无意义测试。

### 10. 多步骤任务必须设检查点
- 超过3步的任务，**每步完成先验证，再继续**。
- 不允许：一步到位写几百行、不验证就往下走。

### 11. 失败要大声（不隐藏错误）
- 报错要明确抛出、日志要完整，**不静默失败、不吞错误**。
- “部分成功”=失败，必须汇报。

### 12. 目录与文件规则（不瞎放）
- 严格按现有目录结构放文件，**不新建目录、不乱放文件**。
- 命名：kebab-case（全小写、连字符），不混用大小写。
